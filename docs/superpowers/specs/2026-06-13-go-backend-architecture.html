DraftRight Backend — Go Clean Architecture

A drop-in Go replacement for the NestJS backend, grown inside the existing draftright-rewrite service. Identical API, same Postgres schema, same JWT — cut over behind Caddy once shadow-compared to parity.

Why

Footprint: 653 MB → ~40 MB

The Node image ships a 279 MB node_modules. A statically-linked Go binary on distroless is ~16× smaller, with lower memory and instant cold start.

Node / NestJS image 653 MB
Go / distroless 40 MB
The shape

Modular — clean architecture inside each module

Top level is sliced by feature (one package per NestJS module), not by layer. Inside every module the dependency rule still holds: domain at the center, handler / repo on the outside, dependencies point inward. A module is self-contained and maps 1:1 to a NestJS module — so the strangler-fig port lifts one folder at a time.

domain.go

the core
  • Entities + business rules
  • Ports (interfaces) it needs
  • Zero framework imports

usecase.go

application
  • Register, CreateCheckout…
  • Orchestrates domain + ports
  • Transaction boundaries

repo_pg.go

data adapter
  • Implements a domain port
  • sqlc on the live schema
  • Swappable / in-memory for tests

adapter/*

external
  • stripe / sepay / resend…
  • One package per provider
  • Behind a domain interface

handler.go

delivery
  • chi routes for this module
  • DTO ⇄ domain mapping
  • Shared mw + error envelope
handler / repo / adapter→usecase→domain(per module · never the reverse)

De-risking the big bang

Strangler-fig cutover via Caddy

Go runs alongside Node. Each path is mirrored to both and response-diffed until clean, then its Caddy route flips. A bad flip is reverted by editing one Caddy block — no redeploy.

🟥 NestJS :3000 — serves every path not yet flipped
🟩 Go :3001 — serves /v1/rewrite today; grows path by path

Guardrails

How it honors Rule #1

Drop-in contract

Identical routes, JSON, status codes, JWT. Clients don't change; parity is testable.

Reuse, don't fork

Extend the working Go service + its adapters. Strategy pattern → Go interfaces, 1:1 with the backend.

No hardcoding

Same .env via typed config; provider keys, URLs, limits stay config-driven.

Extensible

New payment method / LLM provider = one adapter package behind an existing port. No handler edits.

Testable in isolation

domain + usecase unit-tested with in-memory adapters (already the pattern in rewrite-go).

Small, focused units

One folder per module per layer; files stay readable and independently reviewable.
