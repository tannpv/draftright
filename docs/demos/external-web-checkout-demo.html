What's actually happening in production:

  1. Tap "Upgrade to Pro" in the Flutter app → app calls POST /payment/checkout on the NestJS backend with the user's id and the desired plan id.
  2. Backend's LemonSqueezyStrategy.createCheckout calls POST /v1/checkouts on Lemon Squeezy and embeds our reference_code in the checkout's custom data. Returns a hosted-checkout URL.
  3. App opens the URL with the platform's external browser (url_launcher → Safari on iOS / Chrome on Android). Lemon Squeezy renders the page on the right. Google Pay shows up automatically when the user's browser advertises it (Chrome on Android, recent Safari on iOS). No client-side work required from us.
  4. User taps Google Pay → completes biometric / passcode auth → done. Lemon Squeezy fires subscription_payment_success webhook to our backend, our verifier checks the HMAC, looks up our reference_code, marks the subscription active.
  5. User taps the back button → returns to our app. App polls /auth/me on resume; the Pro badge flips on.

Why this satisfies Play Store + App Store policy: the in-app button is an information button that leads to a web page. We never call Google Pay's payment APIs from inside the app, and the in-app UI doesn't show a Buy button that competes with Apple IAP / Play Billing. Apple's 4.8 (Login services) is satisfied separately by the Sign-in-with-Apple addition. Play has been tolerating this "user choice billing" link-out pattern globally since 2024.
